Section 1 — Buffer Disjointness

The Problem: False-Positive Barriers in Pipelines

In multi-buffered pipelines, producer and consumer access different slots of the same shared memory allocation via a dynamic phase counter.

memdesc<3×128×128×f16> allocation
┌──────────┬──────────┬──────────┐
│  Slot 0  │  Slot 1  │  Slot 2  │
└──────────┴──────────┴──────────┘
    ↑ consumer               ↑ producer
    phase % 3          (phase+2) % 3

→ Always different slots, but membar can't tell
Membar inserts __syncthreads() / s_barrier between every producer/consumer pair because it cannot prove the dynamic indices point to different slots.

Section 1 — Buffer Disjointness

Why Membar Cannot Distinguish Slots

Membar tracks AllocationSlice — byte intervals in shared memory. The MemDescIndexOp selects a slot via a dynamic SSA value.

TDM write  → memdesc_index %alloc[%idx_a]
local_load → memdesc_index %alloc[%idx_b]

%idx_a = (phase + 2) % 3
%idx_b =  phase      % 3

Both alias to the same base allocation.
getAllBufferIdsWithAliases() → overlapping
isIntersected() → true → barrier inserted

The gap: Membar reasons about which bytes are touched, but has no reasoning about the values of dynamic indices.


It sees two ops touching the same allocation and conservatively assumes they might access the same slot.


This is correct for safety but produces false positives when the indices are provably different.


Cost: unnecessary s_barrier on every pipeline iteration — stalling all warps.

BufferIndexExpr: Symbolic Index Analysis

Pattern-match the index arithmetic to prove idx_a ≠ idx_b symbolically.

// Decompose each index into:
struct BufferIndexExpr {
  Value base;    // loop-carried phase
  int offset;    // constant offset
  int modulus;   // modulus from remsi
};

// Producer: (phase + 2) % 3
//   → {base=phase, offset=2, mod=3}
// Consumer: phase % 3
//   → {base=phase, offset=0, mod=3}

// Same base, same mod,
// offset 2 ≠ 0 → DISJOINT ✓

Recognized Patterns

  • arith.remsi %v, C — modular wrapping
  • arith.addi %v, C — constant offset
  • select(cmpi sge %v, C), sub, %v — conditional wrap (equivalent to rem)

Properties

  • Works in core membar — all backends
  • No annotations or attributes needed
  • Covers what the Gluon pipeliner produces
  • O(1) per index comparison

Range Analysis vs Relational Analysis

Integer Range Analysis (Non-Relational)

Triton has TritonIntegerRangeAnalysis. It tracks per-value intervals: [lo, hi]

idx_a = phase % 3      → [0, 2]
idx_b = (phase+2) % 3  → [0, 2]

Intervals overlap!
Cannot express: "a ≠ b"

Fundamental limitation: non-relational domains track each variable independently. They cannot express relationships between two variables.

Relational Analysis (What We Need)

We need to reason about the relationship between idx_a and idx_b.

Given:  idx_a = base % 3
        idx_b = (base + 2) % 3
Prove:  idx_a ≠ idx_b

For any base:
  (base%3) ≠ ((base+2)%3)
  because 2 ≢ 0 (mod 3)   ✓

BufferIndexExpr is a lightweight relational analysis — it tracks the shared base and compares constant offsets mod N.

Presburger Arithmetic: What MLIR Offers

What is Presburger Arithmetic?

First-order logic over integers with +, <, ∃, ∀, mod.

No multiplication of variables — but that's fine for index arithmetic.


Key property: decidable — every statement can be mechanically proved or disproved.


MLIR's Standalone Library

mlir/Analysis/Presburger/

  • IntegerRelation — systems of linear constraints
  • PresburgerSet — unions of polyhedra with mod
  • GCD test, Simplex solver, Fourier-Motzkin
  • Already used by Affine dialect for dependence analysis

Expressing Buffer Disjointness

// As Presburger constraints:

// Producer index:
∃ q₁: idx_a = phase + 2 - 3·q₁
       0 ≤ idx_a ≤ 2

// Consumer index:
∃ q₂: idx_b = phase - 3·q₂
       0 ≤ idx_b ≤ 2

// Question: is idx_a = idx_b
// satisfiable for any phase?

// Presburger solver: NO
// → provably disjoint ✓

The solver handles this mechanically — no hand-crafted patterns needed.

Section 2 — Warp-Local Access

The Problem: Unnecessary CTA Barriers

When shared memory is partitioned by warp — each warp accesses its own row range — no cross-warp data dependency exists. But membar doesn't know this.

Shared Memory (one tile, warpsPerCTA = [4, 1])
┌──────────────────────────────────┐
│  Warp 0: rows  0-15              │
├──────────────────────────────────┤
│  Warp 1: rows 16-31              │
├──────────────────────────────────┤
│  Warp 2: rows 32-47              │
├──────────────────────────────────┤
│  Warp 3: rows 48-63              │
└──────────────────────────────────┘
Each warp's region is disjoint
→ no cross-warp dependency
→ CTA barrier is unnecessary

What membar sees

  • TDM write → shared allocation [0, N)
  • local_load ← shared allocation [0, N)
  • Same allocation? Yes
  • → Insert CTA barrier

Reality

  • Warp k writes rows [k·16, (k+1)·16) in slot X
  • Warp k reads rows [k·16, (k+1)·16) in slot Y
  • Warp j (j≠k) never touches warp k's row range in any slot
  • → Barrier is a false positive

4 warps, triple-buffered pipeline. Each warp runs an independent batched WMMA on its own slice.

TDM load: [4 × BLOCK_N, HEAD_SZ]        Batched WMMA view: [4, BLOCK_N, HEAD_SZ]
┌──────────────────────────┐
│ W0: rows [0,    BN)      │  ──→  W0: independent MMA on its own slice
├──────────────────────────┤
│ W1: rows [BN,  2·BN)     │  ──→  W1: independent MMA on its own slice
├──────────────────────────┤
│ W2: rows [2·BN, 3·BN)    │  ──→  W2: independent MMA on its own slice
├──────────────────────────┤
│ W3: rows [3·BN, 4·BN)    │  ──→  W3: independent MMA on its own slice
└──────────────────────────┘
Pipeline (triple-buffered):                       a = i%3,  c = (i+2)%3

  async_wait(6)
  TDM_write(buf=c)           ← writes slot 2     warpsPerCTA = [4, 1]
  # ← barrier inserted here (FALSE POSITIVE)
  local_load(buf=a)          ← reads slot 0      warpsPerCTA = [4, 1]

Slots are different, AND warps access disjoint row ranges in both slots.

Section 2 — Warp-Local Access

Solution: warpsPerCTA Comparison

The Check

Compare warpsPerCTA from writer and reader. If they match, every warp accesses the same row-range partition on both sides.

Writer: tdmGetWarpDistribution → [4, 1]
Reader: encoding.warpsPerCTA   → [4, 1]

normalize([4, 1]) = [4]
normalize([4, 1]) = [4]
[4] == [4] → MATCH → barrier suppressed ✓

Why This Is Safe

All shared memory encodings give every tensor element a unique byte address (one-to-one mapping). If warps own disjoint element sets → they access disjoint address ranges — regardless of encoding (padded, swizzled, linear).

Tensor elements        Encoding         Byte addresses
(warpsPerCTA=[4,1])    (one-to-one)

┌──────────────┐                    ┌──────────────┐
│ W0: rows 0-15│ → unique addrs →  │ W0: 1024 addr│
├──────────────┤   each element     ├──────────────┤
│ W1: rows16-31│ → gets its own  →  │ W1: 1024 addr│
├──────────────┤   address          ├──────────────┤
│ W2: rows32-47│ → (never shared) → │ W2: 1024 addr│
├──────────────┤                    ├──────────────┤
│ W3: rows48-63│ → unique addrs →  │ W3: 1024 addr│
└──────────────┘                    └──────────────┘
Disjoint elements → disjoint addresses
The encoding rearranges where elements go (padding adds gaps, swizzling shuffles), but never puts two elements at the same byte.

Section 2 — Warp-Local Access

Challenge: MemWaitOpTrait Barrier

The warpsPerCTA check runs in the MembarFilterFn callback. But there's a separate codepath that the filter cannot reach:

// Membar.cpp — unconditional, early-return, filter NEVER consulted
if (op->hasTrait<MemWaitOpTrait>() &&
    !containsLocalBarrier(op->getNextNode())) {
    insertBarrier(op, builder);   // CTA-wide barrier — always
    blockInfo->sync();            // clears ALL tracked dependencies
    return;                       // early return — filter bypassed
}

Current Flow (Conservative)

async_copy → blockInfo records write
async_wait → barrier inserted, sync()
             ↑ UNCONDITIONAL — filter never called
local_load → blockInfo is clean, no conflict

Even with matching warpsPerCTA, the barrier is always emitted after async_wait.

Proposed: Defer to Filter

async_copy → blockInfo records write
async_wait → filter present? don't barrier
             pending writes stay in blockInfo
local_load → isIntersected() → filter called
             warpsPerCTA match → suppressed ✓

When a MembarFilterFn is provided, let the filter decide at the read op instead.

This requires refactoring the MemWaitOpTrait handler in common Membar.cpp — not just adding a backend filter clause. The filter is never consulted today because the handler returns early.

Summary

Two Orthogonal Optimizations

Buffer Disjointness (Problem 1)

  • Problem: dynamic buffer slot indices look aliased
  • Solution: BufferIndexExpr — symbolic pattern matching on index arithmetic
  • Why not range analysis: non-relational, can't express a ≠ b
  • Future: Presburger arithmetic for complex patterns (MLIR library ready)

Warp-Local Access (Problem 2)

  • Problem: per-warp partitions treated as aliasing
  • Solution: warpsPerCTA comparison + one-to-one address mapping
  • Extends to: async_copy, local_store/load
  • Challenge: MemWaitOpTrait handler needs refactoring

These compose naturally: BufferIndexExpr proves slot disjointness across buffer slots, warpsPerCTA proves warp disjointness within each slot. Together they eliminate all false-positive barriers in pipelined FA MQA decode.
