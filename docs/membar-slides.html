Section 1 — Buffer Disjointness

The Problem: False-Positive Barriers in Pipelines

In multi-buffered pipelines, producer and consumer access different slots of the same shared memory allocation via a dynamic phase counter.

memdesc<3x128x128xf16> allocation

  +----------+----------+----------+
  |  Slot 0  |  Slot 1  |  Slot 2  |
  +----------+----------+----------+
      ^ consumer               ^ producer
      phase % 3          (phase+2) % 3

  --> Always different slots, but membar can't tell
Membar inserts __syncthreads() / s_barrier between every producer/consumer pair because it cannot prove the dynamic indices point to different slots.

Why Membar Cannot Distinguish Slots

Membar tracks AllocationSlice — byte intervals in shared memory. The MemDescIndexOp selects a slot via a dynamic SSA value.

TDM write  --> memdesc_index %alloc[%idx_a]
local_load --> memdesc_index %alloc[%idx_b]

%idx_a = (phase + 2) % 3
%idx_b =  phase      % 3

Both alias to the same base allocation.
getAllBufferIdsWithAliases() --> overlapping
isIntersected() --> true --> barrier inserted

The gap: Membar reasons about which bytes are touched, but has no reasoning about the values of dynamic indices.


It sees two ops touching the same allocation and conservatively assumes they might access the same slot.


This is correct for safety but produces false positives when the indices are provably different.


Cost: unnecessary s_barrier on every pipeline iteration — stalling all warps.

BufferIndexExpr: Symbolic Index Analysis

Pattern-match the index arithmetic to prove idx_a != idx_b symbolically.

// Decompose each index into:
struct BufferIndexExpr {
  Value base;    // loop-carried phase
  int offset;    // constant offset
  int modulus;   // modulus from remsi
};

// Producer: (phase + 2) % 3
//   --> {base=phase, offset=2, mod=3}
// Consumer: phase % 3
//   --> {base=phase, offset=0, mod=3}

// Same base, same mod,
// offset 2 != 0 --> DISJOINT

Recognized Patterns

  • arith.remsi %v, C — modular wrapping
  • arith.addi %v, C — constant offset
  • select(cmpi sge %v, C), sub, %v — conditional wrap

Properties

  • Works in core membar — all backends
  • No annotations or attributes needed
  • Covers what the Gluon pipeliner produces
  • O(1) per index comparison

Range Analysis vs Relational Analysis

Integer Range Analysis (Non-Relational)

Triton has TritonIntegerRangeAnalysis. It tracks per-value intervals: [lo, hi]

idx_a = phase % 3      --> [0, 2]
idx_b = (phase+2) % 3  --> [0, 2]

Intervals overlap!
Cannot express: "a != b"

Fundamental limitation: non-relational domains track each variable independently. Cannot express relationships between two variables.

Relational Analysis (What We Need)

We need to reason about the relationship between idx_a and idx_b.

Given:  idx_a = base % 3
        idx_b = (base + 2) % 3
Prove:  idx_a != idx_b

For any base:
  (base%3) != ((base+2)%3)
  because 2 is not 0 (mod 3)

BufferIndexExpr is a lightweight relational analysis — it tracks the shared base and compares constant offsets mod N.

Presburger Arithmetic: What MLIR Offers

What is Presburger Arithmetic?

First-order logic over integers with +, <, exists, forall, and multiplication by constants (not variables). Can express mod via exists q: x = n*q + r.

Key property: decidable — every statement can be mechanically proved or disproved.


MLIR's Standalone Library

mlir/Analysis/Presburger/

  • IntegerRelation — systems of linear constraints
  • PresburgerSet — unions of polyhedra with mod
  • GCD test, Simplex solver, Fourier-Motzkin
  • Already used by Affine dialect for dependence analysis

Expressing Buffer Disjointness

// As Presburger constraints:

// Producer index:
exists q1: idx_a = phase + 2 - 3*q1
           0 <= idx_a <= 2

// Consumer index:
exists q2: idx_b = phase - 3*q2
           0 <= idx_b <= 2

// Question: is  idx_a = idx_b
// satisfiable for any phase?

// Presburger solver: NO
// --> provably disjoint

The solver handles this mechanically — no hand-crafted patterns needed.

Section 2 — Warp-Local Access

Problem 2-1: Unnecessary CTA Barriers Between Write/Read Pairs

When shared memory is partitioned by warp, no cross-warp data dependency exists between the writer (TDM copy / async_copy) and the reader (local_load).

Shared memory tile, warpsPerCTA = [4, 1]:

W0: rows 0-15  — writes & reads here only
W1: rows 16-31  — writes & reads here only
W2: rows 32-47  — writes & reads here only
W3: rows 48-63  — writes & reads here only

Target op pairs for barrier suppression:

  • TDM copy → local_load  implemented
  • async_copy → local_load  extensible
  • local_store → local_load  extensible

What membar sees

  • Writer → shared allocation [0, N)
  • Reader ← shared allocation [0, N)
  • Same allocation? Yes
  • → Insert CTA barrier

Reality

  • Warp k accesses rows [k*16, (k+1)*16) in any slot
  • Warp j (j!=k) never touches warp k's row range
  • → Barrier is a false positive

Example: FA MQA Decode with Split-K

4 warps, triple-buffered pipeline. Each warp runs an independent batched WMMA on its own slice.

TDM load: [4 x BLOCK_N, HEAD_SZ]

W0: rows [0, BN)  → independent MMA
W1: rows [BN, 2*BN)  → independent MMA
W2: rows [2*BN, 3*BN)  → independent MMA
W3: rows [3*BN, 4*BN)  → independent MMA
Pipeline (triple-buffered):
  a = i%3,  c = (i+2)%3

  async_wait(6)
  TDM_write(buf=c)     // writes slot 2
                       // warpsPerCTA = [4, 1]

  # <-- barrier inserted here (FALSE POSITIVE)

  local_load(buf=a)    // reads slot 0
                       // warpsPerCTA = [4, 1]
Slots are different (slot 2 vs slot 0), AND warps access disjoint row ranges in both slots. The barrier is a false positive on both counts.

Solution: warpsPerCTA Comparison

The Check

Compare warpsPerCTA from writer and reader. If they match, every warp accesses the same row-range partition on both sides.

Writer: tdmGetWarpDistribution --> [4, 1]
Reader: encoding.warpsPerCTA   --> [4, 1]

normalize([4, 1]) = [4]
normalize([4, 1]) = [4]
[4] == [4] --> MATCH --> barrier suppressed

Why This Is Safe

All shared memory encodings give every tensor element a unique byte address (one-to-one mapping). Disjoint element sets → disjoint address ranges — regardless of encoding.

One-to-one: elements → addresses

W0: rows 0-15
W1: rows 16-31
W2: rows 32-47
W3: rows 48-63

Tensor elements

→
W0: 1024 unique addr
W1: 1024 unique addr
W2: 1024 unique addr
W3: 1024 unique addr

Byte addresses

The encoding rearranges where elements go (padding, swizzling), but never puts two elements at the same byte. Disjoint elements = disjoint addresses.

Section 2 — Warp-Local Access

Problem 2-2: MemWaitOpTrait Unconditional Barrier

The handler inserts a CTA barrier after every async_wait and returns early — isIntersected / filter never consulted. Can we just remove it and let the normal RAW path handle it?

Current Handler

async_copy --> blockInfo: WRITE [0,N)
async_wait --> barrier + sync() + return
              ^ short-circuits everything
local_load --> blockInfo clean, nothing left

Simple rule: wait → barrier, always. Filter never consulted — no way to suppress for warp-local.

Why Not Just Remove It?

Both async copy ops do declare MemWrite<SharedMemory>, and async_wait has no memory effects. So isIntersected would see the RAW. But:

  • Fragile equivalence — same barrier only because all slots currently alias. Once Problem 1 adds slot distinction, isIntersected sees no overlap → no barrier
  • MemAsyncWriteOpTrait — clears async writes from blockInfo (prevents false barriers before wait). With it, isIntersected sees nothing → no barrier

Design Constraint

The two mechanisms work together by design:

MemAsyncWriteOpTrait:
  Prevents false barriers BEFORE async_wait
  (DMA in-flight, CTA barrier can't help)

MemWaitOpTrait handler:
  Provides the real barrier AFTER async_wait
  (DMA complete, cross-warp visibility needed)

Removing one without reworking the other breaks correctness.

Refactoring Options

  • A: Keep handler, add filter override — let MembarFilterFn suppress the post-wait barrier
  • B: Tag async writes for deferred barrier — richer state in blockInfo
  • C: Separate async write tracking — move to syncWriteSlices at wait time, then normal RAW + filter

Goal: let warpsPerCTA filter suppress post-wait barrier for warp-local, keep correctness for cross-warp and non-AMD.

Summary

Two Orthogonal Optimizations

Buffer Disjointness (Problem 1)

  • Problem: dynamic buffer slot indices look aliased
  • Solution: BufferIndexExpr — symbolic pattern matching
  • Why not range analysis: non-relational, can't express a != b
  • Future: Presburger arithmetic (MLIR library ready)

Warp-Local Access (Problem 2)

  • 2-1: write/read pairs — warpsPerCTA comparison + one-to-one mapping
  • Implemented for TDM → local_load; extends to async_copy, local_store/load
  • 2-2: MemWaitOpTrait unconditional barrier — needs careful refactoring to expose to filter
BufferIndexExpr proves slot disjointness across buffer slots. warpsPerCTA proves warp disjointness within each slot. Together they eliminate all false-positive barriers in pipelined FA MQA decode.
